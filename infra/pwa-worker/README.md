# Appio PWA Worker

Cloudflare Worker that serves user-built PWAs from R2 on `*.appiousercontent.com`.

## Architecture

```
GET https://{app_id}.appiousercontent.com/{path}

в”Ӯ 5. Add CSP + security headers       в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

The builder (`python/builder/src/appio_builder/r2.py` and `kv.py`) is the
producer side of this contract:

- Uploads to `{app_id}/v{N}/{rel_path}` with per-object `Cache-Control` and
  `Content-Type` already set (so the Worker just forwards them).
- Atomically swaps `KV[app_id] = "v{N}"` after every R2 upload succeeds.

The Worker is read-only вҖ” it never writes to R2 or KV.

## One-time setup

### 1. Domains

Register `appiousercontent.com` and add it to the same Cloudflare account
as `appio.app`. Confirm DNS is on Cloudflare nameservers.

Add a wildcard DNS record so subdomains route to the Worker:

```
Type:    AAAA
Name:    *
Content: 100::                # placeholder, proxied
Proxy:   ON (orange cloud)
TTL:     Auto
```

(Cloudflare requires *some* origin for proxied wildcards. The placeholder
IPv6 is the documented pattern; the actual response is generated by the
Worker route.)

Apply for inclusion on the **Public Suffix List**:
https://github.com/publicsuffix/list вҖ” this is what guarantees browsers
isolate cookies between `{a}.appiousercontent.com` and
`{b}.appiousercontent.com`. Without PSL, our security still works (we
never set cookies on this domain) but PSL is the proper hardening.

### 2. R2 bucket

```bash
wrangler r2 bucket create appio-pwas
wrangler r2 bucket create appio-pwas-staging
wrangler r2 bucket create appio-pwas-preview   # for `wrangler dev`
```

The bucket is **private** вҖ” it is accessed only via the Worker binding,
never via a public R2 URL or custom domain.

### 3. KV namespace

```bash
wrangler kv namespace create VERSION_KV
wrangler kv namespace create VERSION_KV --preview
wrangler kv namespace create VERSION_KV --env staging
```

Copy the resulting IDs into `wrangler.toml` (replace the
`REPLACE_WITH_*_NAMESPACE_ID` placeholders).

### 4. Worker secrets

The Worker doesn't currently need any secrets вҖ” bindings cover both R2
and KV. If we add origin auth later (signed URLs, internal-only debug
endpoints) put them in via `wrangler secret put`.

## Local development

```bash
cd infra/pwa-worker
npm install
npm run dev          # spins up Miniflare on http://localhost:8787
```

To exercise the Worker locally you need to populate the preview KV
namespace and preview R2 bucket. Quick smoke test:

```bash
# Seed a fake app
wrangler kv key put --binding VERSION_KV --preview "smoke-test" "v1"

echo '<!DOCTYPE html><html><body>hello</body></html>' > /tmp/index.html
wrangler r2 object put appio-pwas-preview/smoke-test/v1/index.html \
  --file /tmp/index.html \
  --content-type "text/html; charset=utf-8" \
  --cache-control "no-cache, must-revalidate"

# Override the Host header so the Worker sees a subdomain
curl -H "Host: smoke-test.appiousercontent.com" http://localhost:8787/
```

## Deploy

```bash
npm run deploy            # production
npm run deploy:staging    # staging (*.staging.appiousercontent.com)
```

After the first production deploy, verify:

```bash
# 1. KV pointer is read
curl -I https://{some-real-app-id}.appiousercontent.com/

# 2. Security headers are present
curl -sI https://{some-real-app-id}.appiousercontent.com/ | grep -i \
  -E 'content-security-policy|strict-transport|x-frame|x-content-type'

# 3. SPA fallback works
curl -I https://{some-real-app-id}.appiousercontent.com/some/deep/route
# вҶ’ 200, Content-Type: text/html (served from index.html)

# 4. Unknown app returns 404 (not 500)
curl -I https://nonexistent-app-12345.appiousercontent.com/
```

## Cache behavior

| Asset                          | Worker Cache API | Browser Cache-Control                   |
|--------------------------------|------------------|-----------------------------------------|
| `index.html`                   | вқҢ skipped       | `no-cache, must-revalidate` (from R2)   |
| `sw.js` / `service-worker.js`  | вқҢ skipped       | `no-cache, must-revalidate` (from R2)   |
| `manifest.json`                | вқҢ skipped       | `no-cache, must-revalidate` (from R2)   |
| Hashed assets (`app-abc.js`)   | вң… cached        | `public, max-age=31536000, immutable`   |

The Cache API key embeds the version (`{app_id}/v{N}/...`), so version
bumps publish atomically without needing an explicit cache purge: the
new version writes a new cache key, and the old keys are evicted by LRU.

## Failure modes

| Scenario                              | Response | Notes                                       |
|---------------------------------------|----------|---------------------------------------------|
| Host doesn't match `*.appiousercontent.com` | 404 | Apex / multi-level / unrelated domain     |
| `app_id` fails DNS-label regex        | 404      | Defense in depth vs. path injection        |
| `KV[app_id]` missing                  | 404      | App was deleted or never published         |
| `KV[app_id]` malformed                | 404      | Logged; investigate the builder            |
| R2 object missing, no SPA fallback    | 404      | Hashed asset that was deleted              |
| R2 object missing, SPA fallback hit   | 200      | Served `index.html`                        |
| Method not GET/HEAD                    | 405      | `Allow: GET, HEAD`                         |
