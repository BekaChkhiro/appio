# T2.3 — Mobile Validation Runbook

Step-by-step protocol for collecting the empirical numbers that drive the Convex-on-mobile Go/No-Go decision in [`docs/adr/002-convex-mobile-validation.md`](../adr/002-convex-mobile-validation.md).

> **Audience:** the human running the experiment. Every command is meant to be copy-pasted; every observation has a slot in the report.

## 0. One-time setup

```sh
cd apps/mobile-convex-poc
npm install
npm run setup           # builds template, runs `cap add ios` + `cap add android`, syncs
```

If `setup` aborts on `REPLACE_ME` config: edit `packages/templates/todo-list-convex/src/config/firebase.ts` and `…/convex.ts` with real credentials, then re-run.

Verify the harness opens in both IDEs at least once before starting scenarios:

```sh
npm run open:ios        # Xcode → choose iPhone 15 Pro simulator → Run (⌘R)
npm run open:android    # Android Studio → choose Pixel 8 API 34 emulator → Run
```

The first launch on each platform installs Pods / downloads SDK components — bake that latency in before you start the clock.

## 1. Enable instrumentation

The debug overlay only renders when explicitly enabled. After the app loads on the simulator:

1. Sign in with Google.
2. Tap into the URL bar (devtools / Safari → Develop → Simulator → Tasks) and append `?debug=1`. The flag persists in localStorage so it survives WebView reloads.
3. Reload the app — you should see a small dark card in the bottom-right showing `ws: open`, reconnect count, and mutation totals.

For physical devices: append `?debug=1` to the hosted Convex sandbox URL during a one-time staging test, then run `npm run build` so the new bundle ships with the flag already set in localStorage. (Alternative: open `chrome://inspect` for Android, Safari → Develop for iOS, and run `localStorage.setItem("appio:t2.3:debug","1")` directly.)

## 2. Reset before every scenario

In the overlay, tap **reset**. This clears both connection log and audit log so each scenario produces a clean dataset.

## 3. Scenarios

> Capture two numbers per scenario: **outcome** (pass / fail / partial) and the **JSON dumps** from the overlay's `copy conn` + `copy audit` buttons. Paste both into the report alongside the scenario.

### 3.1 Cold launch — connection establishment latency

**Goal:** Time from app launch to first reactive `listTasks` result, on a warm cache vs cold.

iOS:

```sh
xcrun simctl boot "iPhone 15 Pro"
xcrun simctl launch "iPhone 15 Pro" app.appio.convexpoc
# In overlay: reset, then force-quit + relaunch the app from the home screen.
```

Android:

```sh
adb shell am force-stop app.appio.convexpoc
adb shell am start -n app.appio.convexpoc/.MainActivity
```

Record:
- `transitions[0]` (first `to: "connected"`) → `deltaMs` is your cold-launch reconnect time.
- Time-to-first-task-render (compare wall clock to your tap on the launch icon).

### 3.2 Airplane mode while idle

**Goal:** Verify the WebSocket reconnects cleanly when network returns, with no manual user action.

iOS Simulator does **not** simulate airplane mode at the system level. Use the **Network Link Conditioner** workaround (`Settings → Developer → Network Link Conditioner → 100% Loss`) and toggle the profile in/out:

```sh
# Pre-condition: app is open + signed in, overlay shows ws: open.
# Manual: Settings → Developer → Network Link Conditioner → ON (100% Loss)
# Wait 30 s — overlay should flip to ws: closed within ~5 s.
# Manual: Network Link Conditioner → OFF
# Watch overlay: ws should re-open. The most recent transition's deltaMs is your reconnect latency.
```

Android Emulator supports a real airplane-mode toggle:

```sh
# In the emulator's extended controls (toolbar → ⋯ → Cellular → Network type: None)
# OR via adb on Android 12+:
adb shell svc wifi disable
adb shell svc data disable
sleep 30
adb shell svc wifi enable
adb shell svc data enable
```

Record: reconnect deltaMs, whether `useQuery` returned fresh data automatically (should — this is the headline behaviour).

### 3.3 Airplane mode with queued mutations

**Goal:** Verify Convex's internal mutation queue replays writes after reconnect, without data loss.

Steps:

1. Reset overlay logs.
2. Disconnect (per 3.2).
3. While disconnected, perform exactly **5 mutations**: 2 creates, 2 toggles, 1 delete. Note the local time you started.
4. Each tap should produce an instant optimistic UI update (the audit overlay's `pending` count increments). If it does NOT, log that as a regression — optimistic patches are supposed to render even when the WS is closed.
5. Wait 60 s with the app foregrounded, still offline.
6. Reconnect.
7. Capture: how long until `pending → 0`? Are all 5 audit entries `acked` (with non-null `ackedAt`)?

Pass criterion: zero failed mutations, zero lost entries (every audit row eventually gets an `ackedAt`), drain time within 5 s of the first reconnect transition.

### 3.4 App backgrounded across reconnect

**Goal:** WebViews on iOS get aggressively reaped when an app backgrounds. Verify Convex resumes correctly when the user comes back.

1. Reset logs.
2. Open the app, sign in, observe `ws: open`.
3. Press home / swipe up to background the app. Wait 60 s.
4. Foreground the app.
5. Capture: did the JS context survive (audit log still populated) or get torn down (audit log empty)? If torn down, capture the new cold-launch reconnect latency.

This is the worst-case for any web-backed mobile app. Record both iOS and Android.

### 3.5 WiFi → cellular handoff (physical device only)

**Goal:** Mobile users walk between WiFi networks and LTE all day. Validate the WS survives.

Requires deploying the harness via TestFlight / Internal Testing track, OR running the bundle from a Mac dev server reachable on cellular (port-forward via ngrok). Capacitor's `server.url` config makes the second path one line.

1. Connect device to home WiFi. Open app, sign in.
2. Tap-spam the create button to put 10 mutations through (so we have something to compare).
3. Walk outside / disable WiFi → device falls back to cellular.
4. After 30 s on cellular, perform 3 more mutations.
5. Reconnect to WiFi.
6. Capture transitions log + audit log.

Pass criterion: zero failed mutations across the handoff. Reconnect latency under 3 s on each handoff.

### 3.6 Firestore baseline comparison

**Goal:** Per the T2.3 acceptance criteria, prove Convex offline UX is at parity-or-better than the legacy Firestore template.

1. Build the legacy template: `cp -R packages/templates/todo-list /tmp/firestore-baseline/` and run the same offline scenarios in a browser (Firestore's persistence already supports offline writes).
2. Capture: optimistic latency (time from tap to UI update), reconnect time, mutation drain time.
3. Compare against the Convex numbers row-for-row in the report.

If this baseline can't be built easily (Firebase project missing, etc.), skip it but record "no Firestore baseline available — comparing against generic IndexedDB-queue expectation" in the report.

## 4. Capturing logs

For each scenario:

1. In overlay → **copy conn** → paste into report under "Connection log".
2. In overlay → **copy audit** → paste into report under "Mutation audit".
3. Take a screenshot of the final UI state for visual comparison.

## 5. After all scenarios complete

Compute the headline numbers required by the acceptance criteria:

- Reconnect latency: median + p90 across scenarios 3.2, 3.3, 3.4, 3.5.
- Lost-mutation rate: `failed / total` across scenarios 3.3, 3.4, 3.5.
- Time-to-first-render: average across cold launches in 3.1 + 3.4.
- WebSocket-survives-handoff: bool from 3.5.

Fill these into the **Headline Numbers** table in the report, then write the **Decision** section using the criteria already laid out there.

## Common gotchas

- **Convex codegen never ran**: the build will fail at `import { api } from "../convex/_generated/api"`. Run `npx convex codegen --typecheck=disable` once in the staging dir (the build script does this).
- **iOS Simulator can't toggle airplane mode**: use Network Link Conditioner (see 3.2). Don't rely on `Settings → Airplane Mode` — that toggle is greyed out on simulators.
- **WebView reaped during background**: not a bug, that's how iOS works. Capture it as part of the data and the report's recommendation will discuss whether to add a Capacitor plugin (`@capacitor/app` `appStateChange` event) to gracefully resume.
- **Clipboard copy doesn't work on file://**: the overlay falls back to a hidden textarea + `execCommand("copy")`, which works in WKWebView. If clipboard is still empty, open Safari → Develop → Simulator → Tasks → Console and run `console.log(window.localStorage.getItem("appio:t2.3:connection-log"))`.
